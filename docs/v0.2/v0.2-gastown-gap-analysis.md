## 0. 本文定位

`v0.2-draft.md` 从 HoC 自身出发，识别了三条"最后一公里"断层（AI 行为规范、自动化闭合、信息质量），定义了方向 A/B/C。

本文从**外部参照系**出发——对 Gas Town 的 849 个 Go 源文件（315,786 行）做逐模块映射，找出 `v0.2-draft.md` 未覆盖但对 v0.2 有价值的补充方向。两份文档合并阅读，形成完整的 v0.2 决策输入。

## 1. 量化差距全景

### 1.1 硬数字

| 指标 | Gas Town | House of Cards | 倍数 |
|------|----------|---------------|------|
| Go 源文件 | 849 | 52 | 16x |
| Go 总行数 | 315,786 | 14,538 | **22x** |
| 内部包 (`internal/`) | 63 | 10 | 6x |
| CLI 命令文件 | 320 | 20 | 16x |
| 测试文件 | 366 | 11 | 33x |
| 外部依赖 | 20+ | 6 | 3x |
| CI/CD Workflows | 10+ | 0 | — |

### 1.2 代码量差距的成因分解

22 倍的差距不全是功能差距。拆解如下：

```
├── ~40%  生产化打磨
│         ├── 366 个测试文件（vs HoC 11 个）
│         ├── 跨平台兼容（Windows、FreeBSD、darwin）
│         ├── 10+ GitHub Actions workflows
│         ├── GoReleaser + npm + Homebrew 分发
│         └── 版本检测、升级提示、诊断工具
│
├── ~30%  重型基础设施的胶合代码
│         ├── doltserver/ — Dolt 数据库进程管理
│         ├── beads/ — beads SDK 集成、路由、类型
│         ├── tmux/ — tmux 会话管理、健康检查
│         ├── connection/ + keepalive/ — 连接管理
│         ├── lock/ — 分布式锁
│         └── daemon/ + boot/ + deacon/ — 三层守护链
│
└── ~30%  真正的功能差距
          ├── swarm/ — 大任务自动拆分
          ├── scheduler/ — 容量感知调度
          ├── feed/ — 实时活动流
          ├── plugin/ — 可扩展插件系统
          ├── quota/ — API 调用限额
          ├── krc/ — 跨会话知识保留
          └── 等（详见第 2 节）
```

**结论**：HoC 保持当前轻量架构，即使补齐所有核心功能并完成生产化打磨，预期代码量在 **4–5 万行**（Gas Town 的 1/6），而非追到 31 万行。这是架构选择（SQLite vs Dolt、单一 Whip vs 三层守护）带来的结构性红利。

## 2. 功能差距逐模块映射

以下将 Gas Town 的 63 个内部包逐一映射到 HoC，分为三个梯队。

### 2.1 第一梯队 — 核心缺失（建议 v0.2 评估）

这些功能直接影响 HoC 在真实多 Agent 场景中的可用性。

| Gas Town 模块 | 功能 | HoC 现状 | v0.2-draft 是否已覆盖 | 建议 |
|--------------|------|---------|---------------------|------|
| `swarm/` | 将 Epic 拆成多个并行子 Bill，跟踪整体进度 | 无 | B-3 部分涉及（复杂度拆分建议） | **扩展 B-3**：不仅建议拆分，还要有 `hoc bill split` 的执行路径 |
| `scheduler/capacity/` | 基于 Minister 技能 + 负载的容量感知调度 | Whip 做简单匹配 | A-2 涉及 Autoscale | **补充**：Autoscale 需要容量模型支撑，否则只是盲目扩缩 |
| `mail/` | 复杂路由、优先级、投递确认 | Gazette 替代 | B-1/B-2 涉及 Gazette 增强 | **评估**：Gazette 是否需要投递确认机制？目前无法知道 Gazette 是否被读取 |
| `convoy/` | Convoy 状态机（编队、就绪、分发、完成检查） | Session 概念覆盖 | 未覆盖 | **评估**：Session 是否需要更丰富的状态机（不仅 open/closed） |

#### 建议新增方向 D：任务编排增强

```
D-1  Bill 拆分执行（扩展 B-3）
     hoc bill split <id> --into <子bill1> <子bill2> ...
     自动创建依赖关系，原 Bill 变为 Epic 状态

D-2  Session 状态机增强
     open → planning → active → reviewing → merging → closed
     每个阶段有明确的进入/退出条件

D-3  Gazette 投递确认
     gazettes 表增加 delivered_at / read_at
     Whip 可检测"未读积压"并告警
```

### 2.2 第二梯队 — 进阶能力（建议 v0.3 评估，v0.2 预留接口）

这些功能在规模化时（5+ Minister 并行）才成为痛点。

| Gas Town 模块 | 功能 | HoC 现状 | 建议 |
|--------------|------|---------|------|
| `feed/` | 实时活动流 + Curator 整理 | 无 | v0.2 可在 `hoc floor` TUI 中预留 feed 面板位置 |
| `plugin/` | 可扩展插件（门控：cooldown/cron/condition/event） | Formula 部分覆盖 | v0.2 的 Formula 系统已足够，v0.3 考虑 trigger 扩展 |
| `quota/` | API 调用限额管理 | 无 | v0.2 暂不需要；多 Provider 时重要 |
| `krc/` | Knowledge Retention Cache — 跨会话知识保留 | `agent_context.md` 手动维护 | v0.2 可将 agent_context 机制化（自动提取 Hansard 关键信息） |
| `dog/` | 工作成果自动验证（构建、测试、lint） | Committee 审查部分覆盖 | v0.2 的 Committee 可增加自动化检查脚本钩子 |
| `checkpoint/` | 中间状态快照 | 无 | v0.3；当前 Git commit 足够 |
| `nudge/` | Agent 卡住时主动干预 | Whip stuck 检测覆盖 | 已有，可在 v0.2 增强干预手段 |

#### v0.2 可做的预留

### 2.4 HoC 不需要的 Gas Town 模块

以下模块是 Gas Town 重型架构的产物，HoC 的轻量设计天然不需要：

| Gas Town 模块 | 不需要的原因 |
|--------------|-------------|
| `doltserver/` | HoC 用 SQLite，无需管理外部数据库进程 |
| `beads/` | HoC 自建 store，不依赖外部 issue 追踪 |
| `connection/` + `keepalive/` | SQLite 无连接管理需求 |
| `lock/` | 单进程 SQLite WAL 模式，无分布式锁需求 |
| `boot/` + `deacon/` | 单一 Whip 已覆盖 |
| `mq/` | 无消息队列需求 |
| `wasteland/` | 无对应概念 |

## 3. 对 v0.2-draft.md 的补充建议

### 3.1 已有方向的增强

| v0.2-draft 方向 | 补充建议 | 依据 |
|-----------------|---------|------|
| **A-2 Autoscale** | 增加简单的容量模型：`pending_bills / (idle_ministers + 1)` 作为扩缩信号，而非仅 `pending > idle * 2` | Gas Town `scheduler/capacity/` 的简化版 |
| **B-1 Gazette 结构化** | 增加 `delivered_at` / `read_at` 字段，Whip 检测未读积压 | Gas Town `mail/` 的投递确认 |
| **B-3 复杂度拆分** | 从"建议拆分"升级为 `hoc bill split` 命令，自动创建子 Bill + 依赖 | Gas Town `swarm/` 的简化版 |
| **C-2 Doctor 增强** | 增加 `hoc doctor --fix` 自动修复模式（清理孤立 worktree、重置卡住的 Minister） | Gas Town `doctor/` 的增强诊断 |
| **C-3 Hansard 回放** | 增加 Minister 能力画像：基于历史 Hansard 的 portfolio 成功率统计 | Gas Town 的 Agent 选择机制 |

### 3.2 建议新增方向

以下方向未在 `v0.2-draft.md` 中出现，但基于 Gas Town 对比认为值得在 v0.2 评估：

#### 方向 D：任务编排增强（与方向 B 互补）

| 编号 | Feature | 价值 | 工作量评估 |
|------|---------|------|-----------|
| D-1 | `hoc bill split` — Bill 拆分执行 | 处理复杂任务的关键能力 | 中（store + cmd） |
| D-2 | Session 状态机扩展（6 阶段） | 让 Session 生命周期可追踪 | 小（store + whip） |
| D-3 | Gazette 投递确认 | 确保信息到达 | 小（store 字段） |

#### 方向 E：生产化基础（v0.2-draft 未涉及）

| 编号 | Feature | 价值 | 工作量评估 |
|------|---------|------|-----------|
| E-1 | GitHub Actions CI（build + test + lint） | 每次提交自动验证，防止回归 | 小（YAML 配置） |
| E-2 | `golangci-lint` 接入 | 代码质量门禁 | 小（配置 + 修复存量 lint） |
| E-3 | `hoc version` 增强 — 构建信息注入 | 问题排查基础 | 小（ldflags） |
| E-4 | Makefile 标准化（build/test/lint/install） | 统一开发者入口 | 小 |

## 6. HoC 的结构性优势（v0.2 应坚守的设计决策）

在补齐功能的过程中，以下 HoC 相对 Gas Town 的优势必须坚守，不能为了补功能而丢掉：

| 优势 | 具体约束 |
|------|---------|
| SQLite 内嵌 | 不引入任何外部数据库进程 |
| tmux 可选 | Runtime 接口保持抽象，新功能不能假设 tmux 存在 |
| 单一 CLI | 不拆分为多个命令行工具 |
| Gazette 压缩 | 不退化为传递原始内容 |
| 零额外进程 | 不引入 Redis、Kafka 等中间件 |
| 隐喻一致 | 新概念必须有政府隐喻对应 |
