## 0. 本文定位

本文是 v0.2 总体草案的**第四个方向**，与已有方向的关系：

| 方向 | 主题 | 本文关系 |
|------|------|---------|
| A | 打通骨架（CLAUDE.md / Autoscale / API） | 无重叠 |
| B | 信息质量（Gazette 结构化） | D-1 与 B-1 互补（B 管格式，D 管事件） |
| C | 可靠性（测试 / doctor / 回放） | D-1 与 C-3 互补（C 管 Hansard 回放，D 管统一事件流） |
| **D** | **运营治理（事件流 / 输入防护 / 治理动作）** | 新增方向 |

方向 D 的核心观点：**HoC 的编排内核已经很强（Whip / Hansard / 质量评分），但"过程可见性"和"治理入口标准化"偏弱。** 这不需要替换内核设计，只需要在现有架构上补充运营层能力。

## 1. 评估方法说明

对比对象为 Edict（三省六部），一个 Python + PostgreSQL + Redis Streams 的多 Agent 编排框架。对比结论基于**源码级审查**，而非文档或 README 宣传。

关键发现：Edict 在架构层面有一些好思路，但多处"已落地"的能力实际上只是 TODO 注释或 prompt 层指令。下文仅保留经过代码验证、确认有实际借鉴价值的项。

## 2. Feature 列表

### D-1：统一事件表（Event Ledger）

**问题**

HoC 当前有两类审计数据：
- **Hansard**：结果级审计（outcome / quality / duration），每个 Bill 完成时写一条
- **Gazette**：通信载体（from / to / summary），投递后标记 read

但缺少**过程级事件记录**：状态变更、by-election 触发、auto-assign 决策、Gazette 投递成功/失败、committee 审查结果等，这些关键动作分散在 slog 日志里，没有结构化持久。

这导致：
1. 故障定位只能靠翻日志文件
2. Session 复盘缺少过程证据（只有最终结果）
3. 无法回答"这个 Bill 为什么被分配给这个 Minister"之类的过程性问题

**设计**

在 SQLite 中新增 `events` 表：

```sql
CREATE TABLE IF NOT EXISTS events (
    id TEXT PRIMARY KEY,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    topic TEXT NOT NULL,          -- 'bill.assigned', 'minister.stuck', 'by_election.triggered' ...
    bill_id TEXT,                 -- 关联 Bill（可选）
    minister_id TEXT,             -- 关联 Minister（可选）
    session_id TEXT,              -- 关联 Session（可选）
    source TEXT NOT NULL,         -- 产生者：'whip', 'speaker', 'cli', 'api'
    payload TEXT DEFAULT '{}'     -- JSON，事件特定数据
);

CREATE INDEX IF NOT EXISTS idx_events_topic ON events(topic);
CREATE INDEX IF NOT EXISTS idx_events_bill_id ON events(bill_id);
CREATE INDEX IF NOT EXISTS idx_events_timestamp ON events(timestamp);
```

**事件 Topic 枚举（初始集）**

| Topic | 触发时机 | Source |
|-------|---------|--------|
| `bill.created` | CreateBill | cli / api |
| `bill.assigned` | autoAssign / manual assign | whip / cli |
| `bill.status_changed` | UpdateBillStatus | whip / cli |
| `bill.enacted` | EnactBillFromDone | whip |
| `minister.status_changed` | UpdateMinisterStatus | whip |
| `minister.stuck` | threeLineWhip 检测到无响应 | whip |
| `by_election.triggered` | byElection 启动 | whip |
| `by_election.completed` | byElection 完成 | whip |
| `gazette.created` | CreateGazette | whip / cli |
| `gazette.delivered` | deliverGazette 成功 | whip |
| `gazette.delivery_failed` | deliverGazette 失败 | whip |
| `session.completed` | 全部 Bill 完成 | whip |
| `privy.merge_success` | privyAutoMerge 成功 | whip |
| `privy.merge_conflict` | privyAutoMerge 冲突 | whip |
| `committee.assigned` | committeeAutomation 分配 | whip |
| `committee.result` | pollReviewFile 结果 | whip |
| `autoscale.triggered` | autoscale 触发 | whip |

**写入方式**

在 `store.go` 中新增：
```go
func (db *DB) RecordEvent(topic, source string, billID, ministerID, sessionID string, payload map[string]any) error
```

在 Whip 的各关键路径中调用。不改变现有逻辑，只在关键动作后追加一行事件记录。

**查询方式**

新增 CLI 命令：
```
hoc events list [--topic <topic>] [--bill <bill-id>] [--minister <minister-id>] [--since <duration>]
hoc events timeline <session-id>    # 按时间线展示该 Session 的所有事件
```

`hoc events timeline` 与已有的 `C-3 Hansard 时间线回放` 互补：C-3 聚焦 Hansard 质量分析，D-1 聚焦完整事件流。

**与方向 B/C 的协同**

- B-1（Gazette 结构化）：Gazette 创建时同步写 `gazette.created` 事件，payload 包含结构化元数据
- C-3（Hansard 回放）：可以从 events 表读取更细粒度的事件，丰富回放内容

**工程量**：小。一张新表 + 一个 RecordEvent 方法 + 在 Whip 各路径中调用。不修改现有逻辑。

**设计原则检查**：
- CLI-first：新增 `hoc events` 命令
- 零额外进程：纯 SQLite，无新依赖
- 隐喻一致：Event Ledger 可映射为"议事日志"（与 Hansard 议事录互补）
- 可测试：RecordEvent 是纯数据库操作，可直接测试

### D-2：Bill 入口校验（Input Guard）

**问题**

Bill 创建时（`hoc bill draft` / `hoc session open` / 未来的 API）没有对标题和描述做任何校验。脏数据（过短标题、含文件路径、纯噪音）会污染后续调度和统计。

Edict 的做法是在 prompt 层约束太子 agent 的输入行为（禁止粘贴路径、URL、元数据），但这只是 prompt 层指令，不是代码级防护。HoC 应该在代码层做真正的校验。

**设计**

在 `store.go` 的 `CreateBill` 之前新增校验函数：

```go
// ValidateBillTitle checks that a bill title meets quality standards.
// Returns an error describing the violation, or nil if valid.
func ValidateBillTitle(title string) error
```

校验规则：

| 规则 | 说明 | 拒绝示例 |
|------|------|---------|
| 长度下限 | >= 4 个 Unicode 字符 | "fix" |
| 长度上限 | <= 120 个 Unicode 字符 | （超长标题） |
| 禁止文件路径 | 检测 `/Users/`、`./`、`C:\` 等模式 | "修复 /Users/foo/bar.go 的 bug" |
| 禁止 URL | 检测 `http://`、`https://` | "看看 https://github.com/..." |
| 禁止系统元数据 | 检测 `session_id`、`message_id`、`Conversation` 等 | "Conversation info session 处理" |
| 禁止纯标点/空白 | 全是标点或空格 | "???" |

**集成点**

- `cmd/bill.go` — `bill draft` 命令
- `cmd/session.go` — `session open` 解析 TOML 中的 Bill 标题
- `cmd/serve.go` — 未来 API 端点

校验失败时返回明确错误信息，提示用户修改。

**CLI flag**：`--force` 可以跳过校验（供特殊场景使用）。

**工程量**：极小。一个纯函数 + 三处调用点。

**设计原则检查**：
- 可测试：纯函数，直接表驱动测试
- CLI-first：在 CLI 层拦截
- 不改变现有数据模型

### D-3：显式治理命令（Governance Actions）

**问题**

HoC 的治理能力在调度层很强（Whip 自动化、Speaker Directive），但从 CLI 层面看，人工干预动作分散且不直观：

- 要暂停一个 Bill，得手动 `hoc bill assign <bill> ""` 清空 assignee
- 要恢复一个 stuck Minister，没有直接命令
- 要强制推进一个 Session，得逐个操作 Bill

缺少一组"治理动词"让人类能快速干预。

**设计**

新增以下 CLI 命令（实现为对现有 Store 操作的封装）：

```
hoc bill pause <bill-id> [--reason <reason>]
    # Bill 状态 → draft，清空 assignee，记录 Event + Gazette

hoc bill resume <bill-id>
    # 将 paused Bill 重新放入 Whip 调度队列（状态 → draft，等待 orderPaper 分配）

hoc minister recover <minister-id>
    # stuck → idle，不触发 by-election，保留当前 Bill 分配
    # 适用于短暂抖动恢复的场景

hoc session advance <session-id>
    # 强制将所有 draft 状态 Bill 标记为可调度（跳过依赖检查）
    # 适用于人工判断依赖已满足但系统未检测到的情况
    # 需要 --force flag 确认

hoc session pause <session-id> [--reason <reason>]
    # 暂停 Session：所有关联 Bill 暂停调度（不影响已在执行的 Minister）
    # Session 状态 → paused

hoc session resume <session-id>
    # 恢复暂停的 Session → active
```

**与 D-1 的协同**

每个治理命令执行时写入 Event（topic: `governance.*`），形成治理动作的审计轨迹。

**隐喻映射**

| 命令 | 隐喻 |
|------|------|
| `bill pause` | 议案搁置（Table a Bill） |
| `bill resume` | 恢复辩论（Resume Debate） |
| `minister recover` | 部长复职 |
| `session advance` | 议长推进议程（Advance the Order） |
| `session pause/resume` | 休会 / 复会（Adjourn / Reconvene） |

**Session 状态机扩展**

```
active → paused → active    （新增 paused 状态）
active → completed
active → dissolved
```

需要在 sessions 表的 status 字段和 Whip 的 `ListActiveSessions` 查询中适配。

**工程量**：中。每个命令本身简单（Store 操作 + Event 记录 + Gazette 通知），但涉及 6 个新命令和 Session 状态机扩展。

**设计原则检查**：
- CLI-first：纯 CLI 命令，API 层后续自然映射
- 隐喻一致：每个命令都有对应的议会隐喻
- 零额外进程：纯数据库操作

## 3. 优先级与实施顺序

```
D-1 统一事件表     [高优先] ← 其他 D 系列的基础设施
  |
  |--- D-2 入口校验 [高优先] ← 独立，可与 D-1 并行
  |
  v
D-3 治理命令       [中优先] ← 依赖 D-1（治理动作需要写事件）
```

建议纳入时机：
- **D-1 + D-2**：与方向 A（打通骨架）同期实施，工程量小、风险低
- **D-3**：与方向 B（Gazette 增强）同期实施，因为 Gazette ACK 协议可能也需要 Session 状态机扩展

## 4. 明确不做的事

以下是对比分析中考虑过但决定不采纳的方向：

| 不做 | 原因 |
|------|------|
| Redis Streams 事件总线 | 违反"零额外进程"原则，SQLite events 表足够 |
| PostgreSQL 迁移 | 过度工程化，SQLite WAL 模式完全满足当前规模 |
| React 实时看板 | 违反 CLI-first 原则，v0.3+ 再考虑 TUI/Web |
| Agent 思考/工具调用级记录 | 存储开销过大，当前关注系统级事件即可 |
| 分阶段恢复（retry -> escalate -> rollback）| HoC 的 grace period + by-election 两阶段机制已经够用，Edict 的分阶段恢复实际上也只是 TODO |
