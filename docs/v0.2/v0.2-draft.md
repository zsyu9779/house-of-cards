## 0. 从 v0.1 到 v0.2：定性

v0.1 的目标是**证明框架可以跑通**——单个 Minister 能干活、多个能并行、崩了能补选、Speaker 能感知全局。这些核心机制都已验证。

v0.2 的目标是**让框架在真实任务上可靠运行**。不追求更多 feature，追求**每个已有 feature 能被真正用起来**，而不只是 demo 能跑。

具体来说，v0.1 存在三条「最后一公里」断层：

1. **AI 不知道规矩**：Minister 被传召时没有明确的行为规范，done-file / Gazette 格式全靠 bill brief 里的一段文字，AI 不一定会遵守。
2. **自动化没有闭合**：Autoscale 只记 log、Webhook 只回 acknowledge、API 里 summon 未实现——平台化能力只有骨架。
3. **信息质量太低**：Gazette 是自由文本，Pipeline 上下文注入只能传原始摘要，缺乏结构，下游 Minister 很难高效利用。

v0.2 就是要把这三条断层打通。

## 2. v0.2 设计方向（候选 Feature）

> 以下按三个方向组织。每个方向下的 Feature 是候选项，待调研后确定是否纳入正式迭代。

---

### 方向 A：打通「最后一公里」（补完 v0.1 遗留骨架）

这些是 v0.1 已有结构但未真正工作的部分，技术风险低，价值确定。

#### A-1：Minister Chamber CLAUDE.md 自动写入

**问题**：当前 `minister summon` 只写 bill brief，没有系统级行为规范文件。Minister（AI）不保证会写 `.done` 文件、不保证遵循 Gazette 格式。这是 done-file 检测可靠性的根本威胁。

**方向**：`buildBillBrief` 在 chamber 里同时写入 `.claude/CLAUDE.md`（对应 Claude Code 的 project-level 指令），内容包括：
- **必须**完成后写 `.hoc/bill-{id}.done`（格式规范）
- **必须**遵循 Gazette 格式（可以给模板）
- **不得**直接 push main 分支
- 当前议案上下文摘要

**待调研**：Claude Code 读 CLAUDE.md 的优先级和文件位置（project-level vs repo-level）。

---

#### A-2：Autoscale 实际触发 Minister 传召

**问题**：`autoscale()` 检测到 `pending bills > idle * 2` 时只发 gazette + 记日志，不实际传召。

**方向**：Autoscale 触发时，从「预备部长池」（一组已 appoint 但 offline 的 minister）中选一个合适的，调用 `claudecode.Summon()`。

**待调研**：
- 预备部长池的管理方式（config 里预定义？动态注册？）
- Autoscale 的上下限参数是否需要暴露到 config.toml

---

#### A-3：API 完善（Minister Summon + Bill 生命周期）

**问题**：API Server 是 Phase 10 的核心卖点，但 minister summon 和 webhook 都是 stub。

**方向**：
- `POST /api/v1/ministers/{id}/summon` — 实现完整的传召逻辑
- `POST /api/v1/bills/{id}/enacted` — 手动 enact
- `POST /api/v1/webhooks` — 至少支持 GitHub push 和 PR merged 事件

**待调研**：Webhook payload 格式（GitHub vs GitLab vs 通用）；API 是否需要认证（Bearer token）。

---

### 方向 B：信息质量提升（Gazette 协议增强）

Gazette 是 House of Cards 的核心差异化点。当前实现是自由文本，设计文档里的结构化格式（决议/变更清单/接口契约）没有被 enforce。

#### B-1：Gazette 结构化格式（Schema Enforcement）

**问题**：free-text summary 导致 pipeline 上下文注入质量差，下游 Minister 读到的信息密度低。

**方向**：为 done-file 定义结构化格式（TOML 或 YAML frontmatter），Whip 解析后：
- 存储到 `gazettes.artifacts` 字段（当前是 free JSON）
- `buildUpstreamGazetteSection` 注入结构化的「接口契约」而非原始摘要

**候选格式**（待定）：
```toml
# .hoc/bill-{id}.done

summary = "JWT 认证 3 个端点已完成"

[contracts]
"POST /api/auth/login" = "{ token: string, refresh: string }"
"POST /api/auth/register" = "{ user_id: string }"

[artifacts]
"api/auth/login.go" = "新增"
"api/auth/middleware.go" = "新增"

[assumptions]
db = "假设 PostgreSQL，MySQL 需修改 migration"
```

**待调研**：AI 能否稳定输出这个格式？是否需要在 CLAUDE.md 里提供示例？

---

#### B-2：Gazette 模板命令

**方向**：`hoc gazette template [--type completion|handoff|help|review|conflict]` 输出对应的 Markdown 模板，Minister 可以直接填写后写入 done-file。

这是 B-1 的辅助工具，降低 Minister 写 Gazette 的认知成本。

---

#### B-3：Bill 复杂度驱动的拆分建议

**问题**：当前 `EstimateBillComplexity` 只在 `hoc bill show` 里展示，没有后续行动。

**方向**：当一个 Bill 的复杂度评分超过阈值（或 failed 超过 N 次），Whip 自动在 Speaker context.md 里写入拆分建议。结合 `hoc bill split` 命令（Speaker 或人类执行），把一个 Bill 拆成多个带依赖的子 Bill。

---

### 方向 C：可靠性与可观测性

这类 Feature 不增加新能力，但让现有能力更可靠。

#### C-1：Whip 核心路径测试补强

**问题**：`internal/whip` 覆盖率 12.9%，`threeLineWhip/pollDoneFiles/autoscale` 等核心路径几乎未覆盖。

**方向**：
- 提取 `threeLineWhip` 的心跳判断逻辑为纯函数（`isOverGracePeriod/isOverStuckThreshold`），单独测试
- `pollDoneFiles` 的文件检测逻辑抽离为纯函数测试
- 补充 `autoscale` 的阈值计算测试

**待考虑**：提取纯函数是代码重构，需要不破坏现有行为。

---

#### C-2：`hoc doctor` 增强

当前 doctor 只检查 DB 连接和工具可用性。

**候选新检查项**：
- SQLite 文件大小（超过阈值警告）
- 孤立 worktree 检测（Chamber 存在但对应 Minister 已 offline）
- Gazette 积压量（未读超过 N 份警告）
- 配置文件合法性校验

---

#### C-3：Hansard 时间线回放

**方向**：`hoc session replay <session-id>` 从 Hansard 重建该会期的完整工作时间线：
- 每个 Bill 的状态变化时间点
- 质量分变化曲线（ASCII 折线图，复用已有 chart.go）
- 补选事件标注

价值：post-mortem 分析和 AI 性能评估。

## 3. 暂不纳入 v0.2 的方向（留给 v0.3+）

| Feature | 排除原因 |
|---------|---------|
| Mesh 拓扑 | 需要循环依赖检测，工程量大，真实需求场景待验证 |
| 多 Speaker 竞标制 | 高度实验性，需要大量 AI 调用，成本高 |
| Web UI（HTMX 前端）| 非核心用例，CLI-first 原则 |
| OTLP 真实 Jaeger 接入 | 运维成本高，框架本身使用场景尚未规模化 |
| Codex/Cursor runtime 真实测试 | 依赖第三方 CLI 稳定性，待框架成熟后专项测试 |

## 4. 待调研问题清单

在正式设计 v0.2 之前，需要调研以下问题：

### AI 行为规范
- [ ] Claude Code 读取 CLAUDE.md 的优先级规则（project-level 文件在 `.claude/` 还是仓库根？）
- [ ] Minister 输出结构化 done-file 的可靠性：AI 能否稳定遵循 TOML 格式？
- [ ] `hoc minister auto` 模式下，如何在 Minister 完成一个 Bill 后自动触发下一个？

### 平台化
- [ ] Autoscale 的预备部长池：是否需要新的数据库表 / config 字段？
- [ ] GitHub Webhook 集成的权威 payload 格式和验证机制（HMAC secret）
- [ ] API 认证：Bearer token 还是 mTLS？使用场景是本地还是远程？

### 信息质量
- [ ] Gazette 结构化格式的具体 schema：TOML frontmatter vs 纯 TOML vs JSON？
- [ ] 复杂度拆分阈值：基于关键词的启发式是否足够，还是需要引入 AI 判断？

## 6. 设计原则（从 v0.1 继承，v0.2 须遵守）

1. **CLI-first**：所有 Feature 先有 CLI 接口，API 是可选的外部访问层
2. **零额外进程**：不引入新的外部服务依赖（不上 Redis、不上 Kafka）
3. **隐喻一致**：所有新概念必须找到政府隐喻对应，或明确说明为何例外
4. **AI 行为规范先行**：每个涉及 AI 的 Feature，先设计好 CLAUDE.md 指令，再设计代码逻辑
5. **可测试优先**：新代码的核心逻辑必须能提取为纯函数并有测试
