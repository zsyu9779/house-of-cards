## 0. 本文定位

本文是 v0.2 总体草案（`v0.2-draft.md`）中 **方向 B（信息质量提升）** 和 **方向 C（可靠性）** 的深化设计。聚焦两个核心组件：

- **Gazette**：从"单向广播"升级为"双向确认 + 多轮质询"
- **Whip**：从"God Object"重构为职责分离的子模块架构

两者紧密耦合——Gazette 是 Whip 最大的输出，Whip 是 Gazette 流转的驱动力。

## 1. Gazette：从公报到对话协议

### 1.1 现状问题

v0.1 的 Gazette 实现与设计文档愿景之间存在三个断层：

**断层一：信息结构缺失**

gazettes 表只有 `summary TEXT`（自由文本）和 `artifacts TEXT`（几乎未使用）。设计文档定义的结构化格式（决议/变更清单/接口契约/假设与风险）完全没有被 enforce。所有 Gazette 都是 `fmt.Sprintf` 拼出的硬编码字符串。

代码位置：
- `whip.go` byElection — 手拼 Gazette 字符串
- `whip.go` autoAssign — 手拼 Gazette 字符串
- `store.go` EnactBillFromDone — 手拼 completion Gazette

**断层二：投递模型过于理想**

当前是纯推模式（push-only）：Whip 投递 → 标记 read → 结束。隐含假设是"单次投递即完成信息传递"，但现实中：

- 上游的信息压缩不可能完美——总有下游需要但上游没想到的细节
- 下游拿到 Gazette 后没有任何反馈通道
- 上游不知道下游是否真正理解了

这等同于假设每个 Minister 都是完美的文档作者——对 AI Agent 来说，这个假设风险极高。

**断层三：投递通道不可靠**

`deliverGazette` 的两个策略都有问题：
- **tmux send-keys**：直接往 tmux session 打文本，可能被 Agent 当成 stdin 输入导致意外行为
- **inbox 文件写入**：写了文件但没有机制保证 Minister 会读

### 1.2 核心设计：ACK + Question Time

#### 设计原则

> **Gazette 不是广播，是需要确认的公报。部长发完公报不能走，必须等签收。**

在议会隐喻中，这映射到 **Question Time（质询时间）**：部长发表声明后，其他议员可以提出质询，部长必须回答。

#### 新增概念

| 概念 | 隐喻 | 说明 |
|------|------|------|
| ACK 确认 | 公报签收 | 下游确认"已阅，无疑问，准备开始" |
| Question | 质询 (Question Time) | 下游对公报内容提出疑问 |
| Answer | 部长回答 (Minister's Answer) | 上游回答质询 |
| Thread | 质询线程 | 同一轮质询对话的 Gazette 链 |

#### Gazette 生命周期（修订）

v0.1：
```
created → read → (结束)
```

v0.2：
```
created → delivered → ack                                (零质询，理想情况)
                    → questioned → answered → ack        (一轮质询后确认)
                                            → questioned (追问，多轮)
                    → escalated                           (升级给 Speaker)
```

#### 数据模型变更

```sql
-- gazettes 表新增字段
ALTER TABLE gazettes ADD COLUMN ack_status TEXT DEFAULT '';
  -- '' (未处理) | 'delivered' | 'ack' | 'questioned' | 'escalated'

ALTER TABLE gazettes ADD COLUMN thread_id TEXT DEFAULT '';
  -- 对话线程 ID，同一轮质询共享

ALTER TABLE gazettes ADD COLUMN reply_to TEXT DEFAULT '';
  -- 回复哪条 Gazette（形成对话链）
```

一次完整的质询对话在数据层面是一组 Gazette 的链表：

```
gazette-001 (completion, A→B, thread: t-001)
  └─ gazette-002 (question, B→A, thread: t-001, reply_to: gazette-001)
      └─ gazette-003 (answer, A→B, thread: t-001, reply_to: gazette-002)
          └─ gazette-004 (ack, B→A, thread: t-001, reply_to: gazette-003)
```

#### Gazette 类型扩展

在 v0.1 的 5 种类型基础上新增：

| 类型 | 方向 | 触发场景 |
|------|------|---------|
| `question` | 下游 → 上游 | 下游 Minister 对公报有疑问 |
| `answer` | 上游 → 下游 | 上游 Minister 回答质询 |
| `ack` | 下游 → 上游 | 确认公报信息充分 |

所有类型应注册为 Go 常量枚举，`CreateGazette` 时校验合法性：

```go
const (
    GazetteCompletion = "completion"
    GazetteHandoff    = "handoff"
    GazetteHelp       = "help"
    GazetteReview     = "review"
    GazetteConflict   = "conflict"
    GazetteQuestion   = "question"
    GazetteAnswer     = "answer"
    GazetteAck        = "ack"
    GazetteAutoscale  = "autoscale"
)
```

### 1.3 Gazette 结构化 Payload

#### 动机

自由文本 summary 导致 pipeline 上下文注入质量差。`buildUpstreamGazetteSection` 把上游的原始 summary 全文注入下游 prompt，信息密度低且不可预测。

#### 结构化格式

为 done-file 定义结构化格式（TOML），Whip 解析后存入 `gazettes.payload` 字段：

```toml
# .hoc/bill-{id}.done

summary = "JWT 认证 3 个端点已完成"

[contracts]
"POST /api/auth/login" = "{ token: string, refresh: string }"
"POST /api/auth/register" = "{ user_id: string }"

[artifacts]
"api/auth/login.go" = "新增"
"api/auth/middleware.go" = "新增"

[assumptions]
db = "假设 PostgreSQL，MySQL 需修改 migration"
rate_limiting = "未实现"
```

DB 层新增字段：

```sql
ALTER TABLE gazettes ADD COLUMN payload TEXT DEFAULT '';
  -- JSON 格式的结构化数据，从 done-file TOML 解析而来
```

`buildUpstreamGazetteSection` 改为优先从 payload 提取 contracts 字段注入，回退到 summary 原文。

#### Go 结构体

```go
type GazettePayload struct {
    Summary     string            `json:"summary"`
    Contracts   map[string]string `json:"contracts,omitempty"`
    Artifacts   map[string]string `json:"artifacts,omitempty"`
    Assumptions map[string]string `json:"assumptions,omitempty"`
}
```

### 1.4 Minister 下线条件：ACK 前置

#### 核心规则

> **Minister 完成 Bill 后，必须收到所有下游的 ACK 才能下线。**

这从根本上消除了"上游离线、下游质询无人应答"的问题，不需要任何降级策略。

#### Minister 状态机修订

新增 `briefing` 状态（通报中）：

```
              ┌───────────┐
              │  offline  │
              └─────┬─────┘
                    │ summon
                    ▼
              ┌───────────┐
      ┌────── │   idle    │ ◀──────────────────┐
      │       └─────┬─────┘                    │
      │             │ Whip 分配 Bill            │
      │             ▼                          │
      │       ┌───────────┐                    │
      │       │  working  │                    │
      │       └──┬─────┬──┘                    │
      │          │     │                       │
      │     Bill enacted  stuck/超时            │
      │          │     │                       │
      │          ▼     ▼                       │
      │   ┌──────────┐ ┌───────┐              │
      │   │ briefing │ │ stuck │── Whip 介入   │
      │   └────┬─────┘ └───┬───┘              │
      │        │            │ 补选 / 重启       │
      │   所有下游 ACK       └─────────────────┘
      │        │
      │        ▼
      │   ┌──────┐
      │   │ idle │ (可接新单) 或 offline (无更多工作)
      │   └──────┘
      └─────────────────────────────────────────┘
```

`briefing` 状态的约束：
- **不接受新 Bill 分配**（orderPaper 跳过 briefing 状态的 Minister）
- **必须响应 question Gazette**（这是 briefing 期间的唯一职责）
- **Whip 监控 briefing 时长**（超时触发告警）

#### ACK 模式：Blocking vs Non-blocking

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **Blocking** | 下游 Bill 等待 ACK 后才进入 working | pipeline 拓扑（上游输出是下游输入） |
| **Non-blocking** | 下游 Bill 直接进入 working，ACK 异步 | parallel 拓扑的汇合点 |
| **Auto** | Whip 根据拓扑自动决定 | 默认模式 |

可在 Session 配置中指定：

```toml
[session]
topology = "pipeline"
ack_mode = "auto"  # "blocking" | "non-blocking" | "auto"
```

`auto` 规则：
- pipeline → blocking
- parallel → non-blocking（汇合 Bill 除外）
- tree → 汇合 Bill blocking，其余 non-blocking

### 1.5 文件协议（Minister 侧）

Minister 通过文件系统与 Whip 交互（与现有 done-file 协议保持一致）：

| 文件 | 方向 | 触发 |
|------|------|------|
| `.hoc/bill-{id}.done` | Minister → Whip | Bill 完成（已有） |
| `.hoc/bill-{id}.ack` | 下游 Minister → Whip | 确认 Gazette，准备开始 |
| `.hoc/bill-{id}.question` | 下游 Minister → Whip | 对 Gazette 提出质询 |
| `.hoc/bill-{id}.answer` | 上游 Minister → Whip | 回答质询 |

ACK 文件格式（简单）：

```
ACK
```

Question 文件格式：

```
QUESTION
1. token 过期时间和 refresh 机制是什么？
2. 401 响应时客户端应该怎么处理？
```

Answer 文件格式：

```
ANSWER
1. access token 15min，refresh token 7天，通过 Authorization header 传递
2. 客户端收到 401 应先尝试 refresh，refresh 也失败再跳登录页
```

### 1.6 质询保护机制

防止两个 AI Minister 无限质询形成对话死循环：

```go
const (
    maxQuestionRounds  = 3              // 单个 thread 最多 3 轮质询
    questionTimeout    = 2 * time.Minute // 单轮质询响应超时
    briefingTimeout    = 10 * time.Minute // briefing 状态总超时
)
```

超时/超限处理：
- 单轮超时 → Whip 提醒上游 Minister 尽快回答
- 总轮次超过 3 → 升级给 Speaker 仲裁
- briefing 总时长超过阈值 → Whip 强制 auto-ack + 在 Hansard 写入风险标记

### 1.7 CLAUDE.md 行为规范注入

在 v0.2 A-1（Minister Chamber CLAUDE.md 自动写入）中，加入 ACK 协议规则：

```markdown
## 公报签收协议

当你收到上游议案的公报（Gazette）时：

1. **仔细阅读** 公报中的接口契约和变更清单
2. **评估** 是否有足够信息开始工作
3. 如果信息充分：
   - 写入 `.hoc/bill-{id}.ack`，内容为 `ACK`
4. 如果有疑问：
   - 写入 `.hoc/bill-{id}.question`，格式：
     ```
     QUESTION
     1. 你的第一个问题
     2. 你的第二个问题
     ```
5. 收到回复后重复步骤 2-4，直到信息充分后写 ACK

## 完成后通报协议

当你完成议案后：

1. 写入 `.hoc/bill-{id}.done`（结构化 TOML 格式）
2. 你会进入 **通报状态（briefing）**——此时你不会收到新议案
3. 等待下游 Minister 签收你的公报
4. 如果收到 `.hoc/inbox/` 中的质询公报，请：
   - 阅读质询内容
   - 写入 `.hoc/bill-{id}.answer` 回答所有问题
5. 所有下游签收后，你将恢复为 idle 状态
```

### 1.8 Gazette 投递方式改进

**废弃 tmux send-keys**（风险太高——文本可能被 Agent 当作 stdin 输入）。

改为统一使用 **文件 inbox** 投递：
- Whip 写入 `<chamber>/.hoc/inbox/<gazette-id>.md`
- 同时写入 `<chamber>/.hoc/gazette-signal`（空标记文件）
- Minister 的 CLAUDE.md 中规定：检测到 signal 文件时暂停当前工作，阅读 inbox

Signal 文件在 Minister 读取 inbox 后删除（由 Minister 自行清理，或下次 tick 由 Whip 清理）。

## 2. Whip：从 God Object 到职责分离

### 2.1 现状问题

Whip 是项目中最重的单文件（~1115 行），`tick()` 方法调用 7 个完全不同的职责：

```go
func (w *Whip) tick() {
    w.threeLineWhip()            // 心跳检测
    w.orderPaper()               // DAG 调度
    w.pollDoneFiles()            // 完成信号检测
    w.pollIdleMinisterReassign() // Hook 队列
    w.committeeAutomation()      // 委员会审查
    w.gazetteDispatch()          // 公报投递
    w.autoscale()                // 自动扩缩容
}
```

问题不仅是代码组织——所有逻辑共享同一个 10 秒 tick 间隔，但实际需求频率不同。

### 2.2 子模块拆分方案

```
internal/whip/
├── whip.go          // Whip 主结构 + Run loop + tick 调度
├── liveness.go      // threeLineWhip + byElection + isMinisterAlive
├── scheduler.go     // orderPaper + advanceSession + autoAssign + autoscale
├── poller.go        // pollDoneFiles + pollAckFiles + pollQuestionFiles
│                    // + pollReviewFile + committeeAutomation
├── dispatch.go      // gazetteDispatch + deliverGazette
└── report.go        // Report + fmtSeconds (已相对独立)
```

每个子模块暴露一个方法供 `tick()` 调用，内部逻辑自包含。

### 2.3 tick 增加 ACK/Question 轮询

`tick()` 修订：

```go
func (w *Whip) tick() {
    w.threeLineWhip()
    w.orderPaper()
    w.pollDoneFiles()
    w.pollAckFiles()              // 新增：检测 .ack 文件
    w.pollQuestionFiles()         // 新增：检测 .question / .answer 文件
    w.pollIdleMinisterReassign()
    w.committeeAutomation()
    w.gazetteDispatch()
    w.autoscale()
}
```

#### pollAckFiles

扫描所有处于 briefing 状态的 Minister 的下游 chamber：

```
对于每个 briefing 状态的 Minister M：
  获取 M 完成的 Bill B
  获取 B 的所有下游 Bill（通过 depends_on 反查）
  对于每个下游 Bill 的 assignee Minister D：
    检查 D 的 chamber 中 .hoc/bill-{B.id}.ack 是否存在
    → 存在：标记该 Gazette 为 ack，删除 .ack 文件
  如果所有下游都已 ACK：
    → M 状态从 briefing → idle
    → 记录 briefing 时长到 Hansard
```

#### pollQuestionFiles

```
对于每个 working/briefing 状态的 Minister：
  扫描 chamber/.hoc/bill-*.question 文件
  → 创建 question Gazette，路由到上游 Minister

对于每个 briefing 状态的 Minister：
  扫描 chamber/.hoc/bill-*.answer 文件
  → 创建 answer Gazette，路由到下游 Minister
```

### 2.4 threeLineWhip 检测盲区修复

**当前问题**：`isMinisterAlive` 只检查进程是否存在（tmux session / PID），但如果进程存活而 Agent 实际卡住（如死循环等待 stdin），心跳会被持续更新，永远不会触发 stuck。

**改进方案：输出活跃度检测**

```go
func (w *Whip) isMinisterActive(m *store.Minister) bool {
    // 层级 1：进程存在性检查（现有逻辑）
    if !w.isMinisterAlive(m) {
        return false
    }

    // 层级 2：输出活跃度检查（新增）
    // 对 tmux session 检查最后一次输出时间
    tmuxName := fmt.Sprintf("hoc-%s", m.ID)
    // tmux capture-pane + 比对上次快照
    // 如果 5 分钟内无新输出 → 返回 false
    ...
}
```

这将"进程存活"和"Agent 活跃"区分开来：
- 进程死了 → 直接 stuck
- 进程活着但无输出 → 超过阈值后 stuck

### 2.5 错误处理策略分级

当前大量 `_ = err` 遍布关键路径，需要分级治理：

| 级别 | 处理方式 | 适用场景 |
|------|---------|---------|
| **Critical** | 返回 error + 中断当前处理 | Bill 状态更新、Gazette 创建 |
| **Warning** | slog.Warn + 继续 | 心跳更新、Hansard 写入 |
| **Best-effort** | 静默忽略或 slog.Debug | PID 文件清理、inbox 目录创建 |

重点整改的 Critical 路径：
- `byElection` 中的 `ClearBillAssignment` / `UpdateMinisterStatus`
- `pollDoneFiles` 中的 `EnactBillFromDone`
- `advanceSession` 中的 `AssignBill` / `UpdateBillStatus`
- `pollReviewFile` 中的 `UpdateBillStatus` / `UnassignBill`

### 2.6 tick 超时保护

防止某个子任务阻塞导致整个 tick 停摆：

```go
func (w *Whip) tick() {
    ctx, cancel := context.WithTimeout(context.Background(), 8*time.Second)
    defer cancel()

    w.threeLineWhip(ctx)
    w.orderPaper(ctx)
    // ... 每个子任务接受 context，超时自动放弃
}
```

tick 间隔 10 秒，单次 tick 上限 8 秒，留 2 秒 buffer。

### 2.7 gazetteID 碰撞修复

当前实现：

```go
func gazetteID() string {
    return fmt.Sprintf("gazette-%x", time.Now().UnixNano())
}
```

同一 tick 内多次调用可能产生相同时间戳。改为：

```go
import "crypto/rand"

func gazetteID() string {
    b := make([]byte, 8)
    rand.Read(b)
    return fmt.Sprintf("gazette-%x-%x", time.Now().UnixNano(), b)
}
```

或直接使用 UUID。同样的问题存在于 Hansard ID 生成（`fmt.Sprintf("hansard-%x", time.Now().UnixNano())`）。

### 2.8 autoscale 从空壳到可用

当前 `autoscale()` 的 scale-up 只写 Gazette + log，不实际传召。改进方向：

- 从"预备部长池"（已 appoint 但 offline 的 minister）中选择合适的 Minister
- 调用 `runtime.Summon()` 实际启动
- 阈值参数从 config.toml 读取（不再硬编码）
- Scale-down 时清理 chamber/worktree

此改进与 v0.2-draft 中的 A-2 对齐。

## 3. 质询数据度量：信息反馈闭环

### 3.1 可度量指标

质询过程本身是 Gazette 质量和 Minister 能力的重要信号：

| 指标 | 含义 | 数据来源 |
|------|------|---------|
| **ACK 轮次** | 下游花了几轮才确认 | thread 中 question/answer 数量 |
| **首次 ACK 率** | 零质询直接 ACK 的比例 | 历史 Gazette 统计 |
| **质询响应时间** | 上游收到 question 到写出 answer 的耗时 | Gazette 时间戳差 |
| **briefing 总时长** | 从 enacted 到最后一个 ACK | Minister 状态变化时间戳 |
| **升级率** | 多少质询最终升级到 Speaker | escalated 类型 Gazette 数量 |

### 3.2 Hansard 集成

这些指标写入 Hansard，成为 Minister 的"履历"评分依据：

```go
type Hansard struct {
    // ... 现有字段 ...
    AckRounds    int     // 本次 Bill 的质询轮次
    BriefingDurS int     // briefing 阶段总耗时（秒）
}
```

`FindBestMinisterForSkill` 选人公式扩展：

```go
// 现有：score = quality / float64(load+1)
// 扩展：加入交接效率因子
handoverEfficiency := computeHandoverScore(ministerID) // 基于历史 ACK 轮次、briefing 时长
score := quality * handoverEfficiency / float64(load+1)
```

**直观含义**：一个代码写得好（quality 高）但 Gazette 永远写不清楚（handoverEfficiency 低）的 Minister，不应该被分配到 pipeline 上游。

### 3.3 反哺 Gazette 模板设计

质询数据的更大价值在于**数据驱动的模板演进**：

```
统计发现：
  → 80% 的 question 都在问"接口的错误码是什么"
  → 说明 Gazette completion 模板缺少 error contract 字段
  → 在模板里加上 [error_codes] 段

  → "react" portfolio 的 ACK 轮次远高于 "go"
  → 前端需要的信息（组件 props / 状态管理约定）不在通用模板里
  → 为前端 portfolio 定制 Gazette 模板变体
```

不需要一开始就设计完美的 Gazette 格式。让质询数据告诉你缺什么，持续迭代。

## 4. 完整流程示例

### 4.1 Pipeline 拓扑下的 ACK 流程

```
Session: "Build Auth System"
Topology: pipeline
Bills: auth-api → auth-ui (depends_on: auth-api)

     "QUESTION\n1. token 过期时间？\n2. 401 处理策略？"
   - Whip 检测到：创建 question Gazette → 投递到 Minister-Backend inbox
   - Minister-Backend 阅读质询，写 .hoc/bill-auth-api.answer
   - Whip 检测到：创建 answer Gazette → 投递到 Minister-Frontend inbox
   - Minister-Frontend 确认充分 → 写 .ack
   - Whip 检测到：Minister-Backend → idle

═══ 阶段 4：记录 ═══

9. Hansard 记录：
   - Minister-Backend: ACK 轮次 = 1, briefing 时长 = 3m20s
   - 质量评分调整：考虑首次 ACK 失败，微调 handover 效率分

═══ 阶段 5：下游工作 ═══

10. Bill auth-ui 分配给 Minister-Frontend，开始工作
```

### 4.2 超时降级流程

```
质询轮次超限场景：

Round 1: Frontend asks → Backend answers
Round 2: Frontend asks follow-up → Backend answers
Round 3: Frontend asks again → Backend answers
Round 4: Frontend still unsatisfied → 触发限制

Whip 行为：
  → 创建 escalated Gazette
  → 路由到 Speaker
  → Speaker 决策：
    a. 基于上下文直接回答（充当仲裁）
    b. 判断 Bill 拆解不合理，拆分 auth-api 为更细粒度的子 Bill
    c. 强制 auto-ack + 风险标记，让下游先基于现有信息开工

briefing 超时场景：

Minister-Backend 进入 briefing 后 10 分钟无 ACK
  → Whip 检查原因：
    a. 下游 Minister 尚未阅读 → 提醒下游
    b. 下游 Minister 已发 question 但上游未回答 → 提醒上游
    c. 超过上限 → 强制 auto-ack + Hansard 风险标记
```

## 6. 设计原则（本文遵循）

1. **文件协议优先于进程通信**：Minister 与 Whip 之间的所有交互通过文件系统，不依赖 tmux send-keys 等进程注入手段
2. **隐喻一致**：briefing（通报中）、Question Time（质询）均有明确的政府隐喻映射
3. **渐进增强**：ACK 基础设施先行，多轮质询后加，度量最后——每一步都独立可用
4. **数据驱动**：不追求设计出完美的 Gazette 格式，通过质询数据持续反馈迭代
5. **超时兜底**：所有等待都有上限，防止系统停摆
