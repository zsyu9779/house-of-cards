## 0. v0.2 总体目标

v0.1 证明了框架可以跑通。v0.2 的目标是 **让框架在真实任务上可靠运行**。

v0.1 存在三条核心断层 + 一个运营空白：

1. **AI 不知道规矩** — Minister 被传召时无行为规范，done-file / Gazette 格式靠 brief 文字暗示
2. **自动化没有闭合** — Autoscale 只记 log、Webhook 只回 acknowledge、API 里 summon 未实现
3. **信息质量太低** — Gazette 是自由文本，pipeline 上下文注入密度低，下游难高效利用
4. **过程不可见** — 关键事件散落在 slog 日志中，无结构化审计，治理动作缺乏标准入口

v0.2 围绕 **五个方向** 打通这些断层：

| 方向 | 主题 | 核心价值 |
|------|------|---------|
| **A** | 打通骨架 | 补完 v0.1 遗留的 stub 和空壳 |
| **B** | 信息质量 | Gazette 从广播升级为双向确认协议 |
| **C** | 可靠性与可观测性 | Whip 重构 + 测试补强 + 诊断增强 |
| **D** | 运营治理 | 事件流 + 输入防护 + 显式治理命令 |
| **E** | 生产化基础 | CI / lint / 构建信息 |

## 2. 方向 A：打通骨架

### A-1 Minister Chamber CLAUDE.md 自动写入 [P0]

**问题**：Minister 被传召时没有系统级行为规范文件。AI 不保证写 `.done` 文件、不保证遵循 Gazette 格式，是 done-file 检测可靠性的根本威胁。

**设计**：`buildBillBrief` 在 chamber 里同时写入 `.claude/CLAUDE.md`，内容包括：

- **必须**完成后写 `.hoc/bill-{id}.done`（结构化 TOML 格式，见 B-1）
- **必须**遵循 Gazette 格式模板
- **不得**直接 push main 分支
- 当前议案上下文摘要
- **公报签收协议**（ACK + Question Time 规则，见 B-1）
- **完成后通报协议**（briefing 状态规则，见 B-1）

CLAUDE.md 模板示例见 B-1 的 1.7 节。

**集成点**：`internal/whip/whip.go` — `buildBillBrief` 函数。

---

### A-2 Autoscale 实际触发 + 容量模型 [P1]

**问题**：`autoscale()` 检测到 pending bills 积压时只发 gazette + 记日志，不实际传召。且阈值逻辑过于简陋（`pending > idle * 2`）。

**设计**：

1. **预备部长池**：已 appoint 但 offline 的 minister 构成预备池
2. **容量模型**：扩缩信号改为 `pending_bills / (idle_ministers + 1)`，引入上下限参数
3. **触发传召**：从预备池选合适 Minister，调用 `runtime.Summon()` 实际启动
4. **Scale-down**：清理 chamber/worktree
5. **阈值参数**：从 `config.toml` 读取，不再硬编码

**待定**：预备部长池管理方式（config 预定义 vs 动态注册）。

---

### A-3 API 完善 (Summon + Bill 生命周期 + Webhook) [P1]

**问题**：API Server 的 minister summon 和 webhook 都是 stub。

**设计**：

- `POST /api/v1/ministers/{id}/summon` — 实现完整传召逻辑
- `POST /api/v1/bills/{id}/enacted` — 手动 enact
- `POST /api/v1/webhooks` — 至少支持 GitHub push 和 PR merged 事件（HMAC secret 验证）

**待定**：API 认证方案（Bearer token vs mTLS）；多平台 webhook 格式兼容。

## 3. 方向 B：信息质量提升

### B-1 Gazette 结构化 + ACK 协议 [P0]

这是 v0.2 最大的单项 Feature，从四个维度升级 Gazette 体系。

#### 3.1.1 结构化 Payload

**问题**：free-text summary 导致 pipeline 上下文注入质量差。

**设计**：为 done-file 定义 TOML 结构化格式：

```toml
# .hoc/bill-{id}.done

summary = "JWT 认证 3 个端点已完成"

[contracts]
"POST /api/auth/login" = "{ token: string, refresh: string }"
"POST /api/auth/register" = "{ user_id: string }"

[artifacts]
"api/auth/login.go" = "新增"
"api/auth/middleware.go" = "新增"

[assumptions]
db = "假设 PostgreSQL，MySQL 需修改 migration"
rate_limiting = "未实现"
```

Go 结构体：

```go
type GazettePayload struct {
    Summary     string            `json:"summary"`
    Contracts   map[string]string `json:"contracts,omitempty"`
    Artifacts   map[string]string `json:"artifacts,omitempty"`
    Assumptions map[string]string `json:"assumptions,omitempty"`
}
```

DB 新增字段：`gazettes.payload TEXT DEFAULT ''`（JSON，从 done-file TOML 解析而来）。

`buildUpstreamGazetteSection` 改为优先从 payload 提取 contracts 注入，回退到 summary 原文。

#### 3.1.2 ACK + Question Time 协议

**核心原则**：Gazette 不是广播，是需要确认的公报。

新增概念：

| 概念 | 隐喻 | 说明 |
|------|------|------|
| ACK | 公报签收 | 下游确认"已阅，准备开始" |
| Question | 质询 (Question Time) | 下游对公报内容提出疑问 |
| Answer | 部长回答 | 上游回答质询 |
| Thread | 质询线程 | 同一轮质询对话的 Gazette 链 |

**Gazette 生命周期（修订）**：

```
created → delivered → ack                                (零质询)
                    → questioned → answered → ack        (一轮质询)
                                            → questioned (追问，多轮)
                    → escalated                           (升级给 Speaker)
```

**DB schema 变更**：

```sql
ALTER TABLE gazettes ADD COLUMN ack_status TEXT DEFAULT '';
  -- '' | 'delivered' | 'ack' | 'questioned' | 'escalated'

ALTER TABLE gazettes ADD COLUMN thread_id TEXT DEFAULT '';
ALTER TABLE gazettes ADD COLUMN reply_to TEXT DEFAULT '';
ALTER TABLE gazettes ADD COLUMN payload TEXT DEFAULT '';
```

**Gazette 类型扩展**（新增 3 种）：

```go
const (
    GazetteCompletion = "completion"
    GazetteHandoff    = "handoff"
    GazetteHelp       = "help"
    GazetteReview     = "review"
    GazetteConflict   = "conflict"
    GazetteQuestion   = "question"   // 新增
    GazetteAnswer     = "answer"     // 新增
    GazetteAck        = "ack"        // 新增
    GazetteAutoscale  = "autoscale"
)
```

#### 3.1.3 Minister 状态机：新增 `briefing`

**核心规则**：Minister 完成 Bill 后，必须收到所有下游的 ACK 才能下线。

```
offline → idle → working → briefing → idle/offline
                         → stuck → (补选/重启)
```

`briefing` 约束：
- 不接受新 Bill 分配
- 必须响应 question Gazette
- Whip 监控 briefing 时长（超时告警）

**ACK 模式**（Session 配置）：

| 模式 | 行为 | 场景 |
|------|------|------|
| Blocking | 下游 Bill 等待 ACK 后才 working | pipeline 拓扑 |
| Non-blocking | 下游直接 working，ACK 异步 | parallel 拓扑 |
| Auto | Whip 根据拓扑自动决定 | 默认 |

#### 3.1.4 文件协议 + 投递方式

**Minister 与 Whip 文件交互**：

| 文件 | 方向 | 用途 |
|------|------|------|
| `.hoc/bill-{id}.done` | M → W | Bill 完成（已有） |
| `.hoc/bill-{id}.ack` | 下游 M → W | 确认 Gazette |
| `.hoc/bill-{id}.question` | 下游 M → W | 质询 |
| `.hoc/bill-{id}.answer` | 上游 M → W | 回答质询 |

**投递方式改进**：
- **废弃** tmux send-keys（风险高，可能被 Agent 当 stdin 输入）
- **统一**文件 inbox：`<chamber>/.hoc/inbox/<gazette-id>.md` + `gazette-signal` 标记文件

#### 3.1.5 质询保护机制

防止 AI 无限对话死循环：

```go
const (
    maxQuestionRounds  = 3               // 单 thread 最多 3 轮
    questionTimeout    = 2 * time.Minute  // 单轮响应超时
    briefingTimeout    = 10 * time.Minute // briefing 总超时
)
```

超限处理：升级给 Speaker 仲裁，或 Whip 强制 auto-ack + Hansard 风险标记。

#### 3.1.6 质询度量 + Hansard 集成

质询数据写入 Hansard，成为 Minister 能力评估的信号：

| 指标 | 含义 |
|------|------|
| ACK 轮次 | 下游花了几轮才确认 |
| 首次 ACK 率 | 零质询直接 ACK 的比例 |
| 质询响应时间 | 上游收到 question 到写出 answer 的耗时 |
| briefing 总时长 | 从 enacted 到最后一个 ACK |

`FindBestMinisterForSkill` 选人公式扩展：加入 handover 效率因子。写 Gazette 写不清楚的 Minister 不应被分配到 pipeline 上游。

### B-3 Bill 拆分建议 + `bill split` 执行 [P1]

**阶段一（建议）**：当 Bill 复杂度评分超阈值或 failed 超 N 次，Whip 自动在 Speaker context.md 中写入拆分建议。

**阶段二（执行）**：`hoc bill split <id> --into <子bill1> <子bill2> ...`
- 自动创建子 Bill + 依赖关系
- 原 Bill 变为 Epic 状态（新增状态）
- 子 Bill 全部 enacted 后，Epic 自动 enacted

## 4. 方向 C：可靠性与可观测性

### C-1 Whip 架构重构 + 核心路径测试 [P0]

#### 4.1.1 子模块拆分

当前 `whip.go` ~1115 行 God Object，拆分为：

```
internal/whip/
├── whip.go          // 主结构 + Run loop + tick 调度
├── liveness.go      // threeLineWhip + byElection + isMinisterAlive
├── scheduler.go     // orderPaper + advanceSession + autoAssign + autoscale
├── poller.go        // pollDoneFiles + pollAckFiles + pollQuestionFiles
│                    // + pollReviewFile + committeeAutomation
├── dispatch.go      // gazetteDispatch + deliverGazette
└── report.go        // Report + fmtSeconds
```

#### 4.1.2 tick 增强

- **新增轮询**：`pollAckFiles()` + `pollQuestionFiles()`（配合 B-1）
- **超时保护**：每次 tick 使用 `context.WithTimeout(8s)`，防止子任务阻塞
- **ID 碰撞修复**：`gazetteID()` / `hansardID()` 改用 `crypto/rand` 附加随机字节

#### 4.1.3 threeLineWhip 检测增强

新增"输出活跃度检测"——区分"进程存活"和"Agent 活跃"：
- 进程死了 → 直接 stuck
- 进程活着但 5 分钟无新输出 → 超阈值后 stuck

#### 4.1.4 错误处理分级

| 级别 | 处理方式 | 适用场景 |
|------|---------|---------|
| Critical | 返回 error + 中断 | Bill 状态更新、Gazette 创建 |
| Warning | slog.Warn + 继续 | 心跳更新、Hansard 写入 |
| Best-effort | 静默 / Debug | PID 文件清理、inbox 目录创建 |

重点整改：`byElection`、`pollDoneFiles`、`advanceSession`、`pollReviewFile` 中的 `_ = err`。

#### 4.1.5 核心路径测试补强

- 提取 `threeLineWhip` 心跳判断为纯函数（`isOverGracePeriod` / `isOverStuckThreshold`）
- `pollDoneFiles` 文件检测逻辑纯函数化
- `autoscale` 阈值计算测试
- 目标：`internal/whip` 覆盖率从 12.9% 提升到 40%+

## 5. 方向 D：运营治理

### D-1 统一事件表 (Event Ledger) [P0]

**问题**：Hansard 记录结果，Gazette 记录通信，但过程级事件（状态变更、by-election 触发、auto-assign 决策等）散落在 slog 日志中，无结构化持久。

**设计**：

```sql
CREATE TABLE IF NOT EXISTS events (
    id TEXT PRIMARY KEY,
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    topic TEXT NOT NULL,
    bill_id TEXT,
    minister_id TEXT,
    session_id TEXT,
    source TEXT NOT NULL,     -- 'whip', 'speaker', 'cli', 'api'
    payload TEXT DEFAULT '{}'
);

CREATE INDEX IF NOT EXISTS idx_events_topic ON events(topic);
CREATE INDEX IF NOT EXISTS idx_events_bill_id ON events(bill_id);
CREATE INDEX IF NOT EXISTS idx_events_timestamp ON events(timestamp);
```

**事件 Topic 枚举（初始集）**：

| Topic | 触发时机 | Source |
|-------|---------|--------|
| `bill.created` | CreateBill | cli / api |
| `bill.assigned` | autoAssign / manual | whip / cli |
| `bill.status_changed` | UpdateBillStatus | whip / cli |
| `bill.enacted` | EnactBillFromDone | whip |
| `minister.status_changed` | UpdateMinisterStatus | whip |
| `minister.stuck` | threeLineWhip 检测 | whip |
| `by_election.triggered` | byElection 启动 | whip |
| `by_election.completed` | byElection 完成 | whip |
| `gazette.created` | CreateGazette | whip / cli |
| `gazette.delivered` | deliverGazette 成功 | whip |
| `gazette.delivery_failed` | deliverGazette 失败 | whip |
| `session.completed` | 全部 Bill 完成 | whip |
| `privy.merge_success` | privyAutoMerge 成功 | whip |
| `privy.merge_conflict` | privyAutoMerge 冲突 | whip |
| `committee.assigned` | committeeAutomation | whip |
| `committee.result` | pollReviewFile 结果 | whip |
| `autoscale.triggered` | autoscale 触发 | whip |
| `governance.*` | 治理命令执行 | cli |

**CLI 命令**：

```
hoc events list [--topic <topic>] [--bill <id>] [--minister <id>] [--since <duration>]
hoc events timeline <session-id>
```

**写入方式**：`store.RecordEvent(topic, source, billID, ministerID, sessionID, payload)` — 在 Whip 各关键路径追加调用，不改变现有逻辑。

### D-2 Bill 入口校验 (Input Guard) [P0]

**问题**：Bill 创建时无标题/描述校验，脏数据污染调度和统计。

**设计**：

```go
func ValidateBillTitle(title string) error
```

| 校验规则 | 拒绝示例 |
|---------|---------|
| >= 4 Unicode 字符 | "fix" |
| <= 120 Unicode 字符 | (超长) |
| 禁止文件路径 (`/Users/`, `./`, `C:\`) | "修复 /Users/foo/bar.go" |
| 禁止 URL (`http://`, `https://`) | "看看 https://..." |
| 禁止系统元数据关键词 | "Conversation info session" |
| 禁止纯标点/空白 | "???" |

**集成点**：`cmd/bill.go`、`cmd/session.go`、`cmd/serve.go`。`--force` flag 可跳过。

---

### D-3 显式治理命令 (Governance Actions) [P1]

**问题**：人工干预动作分散，缺乏标准化"治理动词"。

**新增命令**：

| 命令 | 行为 | 隐喻 |
|------|------|------|
| `hoc bill pause <id> [--reason]` | Bill → draft，清空 assignee，记录 Event + Gazette | 议案搁置 (Table a Bill) |
| `hoc bill resume <id>` | paused Bill 重入调度队列 | 恢复辩论 (Resume Debate) |
| `hoc minister recover <id>` | stuck → idle，不触发 by-election | 部长复职 |
| `hoc session advance <id> --force` | 所有 draft Bill 标记可调度（跳过依赖检查） | 推进议程 (Advance the Order) |
| `hoc session pause <id> [--reason]` | 暂停 Session 调度 | 休会 (Adjourn) |
| `hoc session resume <id>` | 恢复 Session | 复会 (Reconvene) |

**Session 状态机扩展**：`active → paused → active`（新增 `paused` 状态）。

每个命令执行时写入 Event（topic: `governance.*`），与 D-1 协同形成审计轨迹。

## 7. 实施路线

### Phase 1 — 地基（D-1 + D-2 + E-1 + C-1 部分）

独立、低风险、为后续 Phase 提供基础设施。

```
D-1 统一事件表        ← 后续所有 Feature 的审计基础
D-2 Bill 入口校验     ← 独立纯函数，可与 D-1 并行
E-1 GitHub Actions CI ← 质量保障门槛
C-1 Whip 子模块拆分   ← 为 Phase 2 的 poller 扩展做准备（纯重构，不改行为）
```

### Phase 2 — 核心升级（A-1 + B-1 + C-1 续）

v0.2 的核心价值交付。

```
A-1 Minister CLAUDE.md 写入     ← B-1 的前置（规则要写进 CLAUDE.md）
B-1 Gazette 结构化 + ACK 协议   ← 分 3 步实施：
  Step 1: 结构化 payload + done-file TOML 解析
  Step 2: ACK 基础设施（briefing 状态 + .ack 文件 + pollAckFiles）
  Step 3: Question Time 多轮对话（.question/.answer + 轮次限制 + 升级路径）
C-1 Whip 测试补强 + tick 增强   ← 配合 B-1 的 poller 扩展
```

### Phase 3 — 补全（A-2 + A-3 + B-2 + B-3 + D-3）

打通自动化闭环 + 治理入口。


## 9. 设计原则（v0.2 须遵守）

1. **CLI-first** — 所有 Feature 先有 CLI 接口，API 是可选外部层
2. **零额外进程** — 不引入 Redis、Kafka 等外部依赖
3. **隐喻一致** — 新概念必须有政府隐喻映射（briefing=通报中、Question Time=质询、Event Ledger=议事日志、Table a Bill=搁置）
4. **AI 行为规范先行** — 涉及 AI 的 Feature，先设计 CLAUDE.md 指令，再设计代码
5. **可测试优先** — 核心逻辑提取为纯函数并有测试
6. **文件协议优先于进程通信** — Minister 与 Whip 交互通过文件系统
7. **超时兜底** — 所有等待都有上限，防止系统停摆
8. **数据驱动** — 不追求完美格式，通过质询数据持续迭代
