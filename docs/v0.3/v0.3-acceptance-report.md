## 5. 完善计划（`v0.3.1` 清尾版本）

为避免 `v0.3.0` 带瑕疵发布，建议开一个 `v0.3.1` 清尾窗口，专注消除 §4 三条阻断项。**不引入任何新 Feature**。

### Phase 4A · 错误处理清尾（B-1） — 0.5 天

**目标**：cmd 层所有 `_ = db.CreateGazette / CreateHansard / RecordEvent / UpdateMinisterStatus / ClearBillAssignment / UpdateBillBranch` 改为显式处理。

**做法**：

1. 新增辅助函数 `cmd/errutil.go`（或复用 `internal/util/`）：

   ```go
   // warnIfErr 记录非致命错误到结构化日志，返回 err 以便链式使用。
   func warnIfErr(op string, err error, attrs ...any) {
       if err != nil {
           slog.Warn(op, append(attrs, "err", err)...)
       }
   }
   ```

2. 全量替换（30 处）：

   ```go
   // before
   _ = db.CreateGazette(g)
   // after
   warnIfErr("create gazette", db.CreateGazette(g), "gazette_id", g.ID)
   ```

3. 验证：
   - `grep -rn "_ = db\." cmd/ | grep -v _test.go` → 零命中
   - `go test ./cmd/... -race` → PASS
   - `golangci-lint run` → EXIT=0

**受影响文件**：`cmd/bill.go`、`cmd/speaker.go`、`cmd/ministers.go`、`cmd/session.go`、`cmd/privy.go`、`cmd/cabinet.go`；新建 `cmd/errutil.go`。

### Phase 4B · 日志出口收敛（B-3） — 0.25 天

**目标**：`internal/speaker/speaker.go:250-251` 两行 `fmt.Printf` 迁出或改为 slog。

**做法**（择一）：

- **方案 A（推荐）**：将 tmux 模式的用户反馈通过 `return` 值或 channel 返回给 `cmd/speaker.go` 调用方打印，`internal/` 只保留 slog.Info。
- **方案 B**：直接改为 `slog.Info("speaker tmux ready", "session", tmuxName)`；反馈文案由 `cmd/speaker.go` 根据 return 打印。

**验证**：`grep -n "fmt.Print" internal/ --include='*.go' | grep -v _test.go | grep -v logger/logger.go` 应无命中（`logger.go` 的命中在注释里）。

### Phase 4C · 覆盖率拔线（B-2） — 2–3 天

**目标**：总覆盖率从 48.0% → ≥ 60%。

**策略**：挑性价比最高的三个低覆盖包补测，优先补**纯逻辑分支**而非 shell-out 路径。

| 包 | 当前 | 目标 | 补测要点 | 估算 |
|----|------|------|---------|------|
| `internal/runtime` | 16.5% | 50% | `claudecode.go / codex.go / cursor.go` 的 tmux 命令构造分支，用 fake exec.CommandContext 注入；错误路径（tmux 不存在、kill-session 失败）表驱动 | 0.75 天 |
| `internal/chamber` | 28.1% | 55% | worktree 创建/销毁/孤儿清理的错误分支；已有 fake git 基础，补失败注入 | 0.75 天 |
| `internal/util` | 30.9% | 70% | `confirm.go` / `dag.go` 的边界输入（空、循环、畸形 JSON）；纯函数，成本低 | 0.5 天 |
| `cmd` | 43.1% | 55% | `floor.go` / `formula.go` / `serve.go` 的 0% 函数补最小冒烟测试 | 1 天 |

**估算带入公式**：
假设总 statement ≈ N，各包 statements 权重按现状估：runtime ≈ 8%、chamber ≈ 10%、util ≈ 4%、cmd ≈ 25%。上面四项提升分别增加 `0.08*(0.50-0.165) + 0.10*(0.55-0.281) + 0.04*(0.70-0.309) + 0.25*(0.55-0.431) ≈ 0.0268 + 0.0269 + 0.0156 + 0.0298 ≈ 9.9 个百分点`。48.0% + 9.9% = **57.9%**，仍差 2 个百分点。

**补救**：在上述基础上再补 `internal/speaker` 的未覆盖分支（当前 54.0%），目标 65%，可再带动全局 ~1.3 个百分点，最终预计 **59–60%**。若仍差临门一脚，再补 `cmd/privy.go` 的 `0%` 分支。

## 6. 风险与权衡

1. **覆盖率 60% 是否值得硬怼？**
   - 当前 48% 的短板集中在 tmux / git worktree / runtime 这类 shell-out 层，真正的业务逻辑（whip / store / minister / config）都已达标。
   - 若团队共识「60% 是导向指标而非硬 gate」，可考虑在 `v0.3-draft §4` 注脚中改为「核心业务包 ≥ 60%，边界 shell-out 包 ≥ 40%」，此时当前 48% 可视为达线。
   - **推荐**：维持原 60% 硬指标，按 Phase 4C 补齐——这是质量强化版的本分。

2. **cmd 层 `_ = db.*()` 真的有危害吗？**
   - 有。`CreateGazette` / `CreateHansard` / `RecordEvent` 失败意味着**治理事件丢失**（审计洞）。当前完全静默，排障时无痕迹。
   - 改造成本极低（0.5 天），收益极高（审计闭环）。**强烈建议纳入**。

3. **是否需要 `v0.3.1` 这个版本名？**
   - 可选：
     - **方案 A**：保留 `v0.3.0` 的 tag 承诺，把 §5 列为 `v0.3.0` 合并前的 release-blocking 工作（不发 `v0.3.1`）
     - **方案 B**：先发 `v0.3.0-rc1`（带瑕疵），完善后正式发 `v0.3.0`
   - **推荐方案 A**——避免「rc 版长期挂着」的技术债。
