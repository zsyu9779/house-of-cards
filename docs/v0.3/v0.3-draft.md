#### 测试质量规范（强制）

所有 v0.3 新增测试必须遵守以下规范：

1. **禁止只测 happy path**：每个被测函数至少覆盖 1 个错误路径
2. **强制 table-driven**：同一函数的多组输入必须用 `[]struct{...}` table-driven 写法
3. **Mock 限制**：只允许 mock 外部依赖（AI runtime、网络），禁止 mock 内部模块（store、config）
4. **断言明确**：禁止 `if err != nil { t.Fatal(err) }` 之后就结束，必须验证返回值的具体内容
5. **命名规范**：`Test<Function>_<Scenario>`，如 `TestDispatch_DAGWithCycle`

### 方向 E：健壮性治理

#### E-1：错误处理治理

**问题**：代码中 50+ 处 `_ = db.*()` 静默吞没错误，覆盖 store、whip、serve 三层。DB 短暂不可用时状态悄悄丢失，无日志、无告警。

**分布**：
- `internal/whip/` — 15+ 处（liveness、poller、dispatch）
- `internal/store/` — 38+ 处（事务操作）
- `cmd/serve.go` — 9 处（API handler 中的 DB 写入）

**典型案例**：
```go
// cmd/serve.go — webhook 创建 bill 后
_ = db.CreateBill(b)        // 失败则 bill 丢失，客户端收到 "ok"
_ = db.RecordEvent(...)     // 审计记录丢失
```

**方案**：
1. **分级处理**：
   - 关键路径（bill 创建/状态变更）：返回错误，中断操作
   - 辅助路径（gazette、hansard 记录）：`slog.Warn` 记录，不中断
   - 只读路径：可保留 `_ =` 但加注释说明原因
2. **逐模块治理**：whip → serve → store，每个模块单独 PR
3. **配合 B-1/B-2 测试**：错误路径必须有测试覆盖

**关键文件**：全局，优先 `internal/whip/` 和 `cmd/serve.go`

---

#### E-1.1 补充：Whip 自动恢复梯度

> 来源：Claude Code 源码分析（三层恢复机制启发）

**现状**：Minister stuck → 直接触发 by-election（重新分配），过于粗暴。

**Claude Code 做法**：三层恢复——先压缩上下文，再反应式压缩，最后加倍 token，全部失败才终止。

**方案**：在 `threeLineWhip()` 中增加恢复梯度，替代直接 by-election：

```
Minister 异常检测
  → 第 1 次（grace period）：发 Gazette 提醒 Minister 做 checkpoint
  → 第 2 次（stuck threshold）：缩减 Bill scope（标记为需人工拆分）
  → 第 3 次（hard limit）：触发 by-election
```

**实现要点**：
- 在 minister 表增加 `recovery_attempts int` 字段
- `threeLineWhip()` 根据 `recovery_attempts` 决定采取哪级恢复动作
- 每次恢复动作通过 Gazette 通知，Hansard 记录
- by-election 时重置 `recovery_attempts`

**关键文件**：`internal/whip/whip.go`、`internal/store/store.go`

---

#### E-1.2 补充：Minister 上下文健康监控

> 来源：Claude Code 源码分析（自动监控消息列表大小 + 阈值触发压缩）

**现状**：Minister 在长任务中 token 耗尽，Whip 无法感知，只能等超时。

**方案**：在 Gazette 协议中增加 `context_health` 字段：

```go
type GazettePayload struct {
    // 现有字段...
    ContextHealth struct {
        TokensUsed    int  `json:"tokens_used"`
        TokensLimit   int  `json:"tokens_limit"`
        TurnsElapsed  int  `json:"turns_elapsed"`
    } `json:"context_health,omitempty"`
}
```

**Whip 侧逻辑**：
- tick 时检查最新 Gazette 的 `context_health`
- `TokensUsed / TokensLimit > 80%` → 发 Gazette 提醒 Minister 做 checkpoint / 总结
- `> 90%` → 标记 Bill 为 at-risk，考虑拆分或 by-election

**关键文件**：`internal/whip/poller.go`、Gazette 解析逻辑

---

#### E-2：配置启动校验

**问题**：`internal/config/config.go` 的 `LoadConfig()` 只做 TOML 语法解析，不验证字段合法性。无效配置在运行时才暴露。

**缺失的校验**：
- Duration 字符串合法性（`HeartbeatInterval`、`StuckThreshold`）
- 数值范围（`MaxMinisters` > 0、scale 阈值合理）
- 必填字段存在性
- 路径可达性（`Home` 目录存在）

**方案**：
```go
func (c *Config) Validate() error {
    // 校验 duration 可解析
    // 校验数值范围
    // 校验必填字段
    // 返回聚合的 multierror
}
```

在 `LoadConfig()` 返回前调用 `cfg.Validate()`。

**关键文件**：`internal/config/config.go`

## 4. 版本 Gate 与验收标准

### 四个硬性 Gate

每个 Phase 完成后必须通过对应 Gate，不通过不进入下一 Phase：

#### Gate 1：错误处理
- [ ] `_ = db.*()` 在关键路径中清零（辅助路径需有 `slog.Warn`）
- [ ] Whip 恢复梯度实现（3 级恢复，非直接 by-election）
- [ ] 配置校验覆盖所有字段，无效配置在启动时报错

#### Gate 2：测试
- [ ] Whip 覆盖率 ≥ 60%
- [ ] Store 覆盖率 ≥ 50%
- [ ] 所有新增测试符合测试质量规范（table-driven、错误路径覆盖）
- [ ] 零 skip 测试（不允许 `t.Skip()` 留桩）

#### Gate 3：Lint
- [ ] `golangci-lint run` 零 error（warning 可接受）
- [ ] gosec 零 HIGH 级别告警
- [ ] CI 中 lint 作为 blocking check

#### Gate 4：日志
- [ ] `fmt.Printf` 在非 CLI 输出路径中清零
- [ ] 关键操作（bill 状态变更、minister 传召/解职、错误恢复）有结构化日志
- [ ] 日志 level 可通过配置或 CLI flag 控制

### 版本验收 Checklist

v0.3.0 发布前必须全部通过：

- [ ] 四个 Gate 全部绿灯
- [ ] 总测试覆盖率 ≥ 60%
- [ ] `go vet ./...` 零错误
- [ ] `golangci-lint run` 零 error
- [ ] `go build ./...` 零 warning
- [ ] `hoc doctor` 全部 PASS
- [ ] 所有 Phase 的 PR 已合并且 CI 通过

## 6. 设计原则

1. **CLI-first**：所有 Feature 先有 CLI，API 是可选层
2. **可测试优先**：新代码核心逻辑必须可测试
3. **零新依赖**：不引入新的外部服务（slog 用标准库）
4. **错误不沉默**：关键路径必须处理错误，辅助路径至少记录日志
5. **质量优先于数字**：测试覆盖率是结果不是目标，关键路径每个分支都必须覆盖

### 架构验证（来源：Claude Code 源码分析）

以下 HOC 设计经 Claude Code 生产级源码验证，方向正确：

| HOC 设计 | Claude Code 对应 | 验证结论 |
|----------|-----------------|---------|
| Gazette 摘要协议 | 子代理只返回摘要给父 Agent | 完全一致 |
| Chamber（git worktree 隔离） | `isolation: 'worktree'` | 完全一致 |
| Minister 禁止递归调用 | 子 Agent 没有 AgentTool | 完全一致 |
| Whip tick 驱动 | Agent Loop `while(true)` | 本质相同 |
| Hansard 审计记录 | JSONL 转录 + 元数据 | 高度对齐 |

当前最大差距在**韧性工程**（恢复策略、上下文监控），已纳入本版本 E-1.1 和 E-1.2。
