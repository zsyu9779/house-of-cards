# Cyber Parliament 设计构想

> AI Agent 协作框架 — 模拟人类社会分工协作的"AI 团队操作系统"
> 
> 讨论日期：2026-02-28
> 参与人：Boss + Copilot

---

## 1. 背景与启发

### 1.1 灵感来源

- **知乎文章**：[OpenClaw + Claude Code 超强教程：一个人就能搭建完整的开发团队](https://zhuanlan.zhihu.com/p/2010127051726792220)
  - 双层架构：编排层（OpenClaw）+ 执行层（Agent）
  - Agent 选择策略：Codex 主力、Claude Code 前端、Gemini 设计
  
- **Gas Town**（[steveyegge/gastown](https://github.com/steveyegge/gastown)）
  - Multi-agent orchestration system for Claude Code
  - 核心概念：Mayor、Rig、Polecat、Convoy、Beads
  - 不足：强绑定 Claude Code、概念太重、线性协作

### 1.2 核心洞察

> **Agent 的困境不是"能力不足"，而是"角色错位"**

- 人们期待 Agent 是"超级员工"大包大揽
- 但现实：即使是真人，也扛不住"全栈自由人"
- **解决思路：模拟人类社会分工协作**

## 2. 核心理念

### 2.1 核心原则

| 人类怎么协作 | AI 应该怎么学 |
|-------------|---------------|
| Leader 拆解任务 | 任务拆解者角色 |
| 专人做专事 | 专家 Agent 池 |
| 问同事比看代码快 | Agent 之间直接沟通 |
| 信息经过凝练 | Agent 间传摘要，不传原始数据 |

### 2.2 设计目标

> 不是做一个"更聪明的 Agent"，而是做一个**AI 团队操作系统**

- 有角色分工
- 有信息凝练
- 有协作接口
- 协作模式可配置

## 3. 架构设计

### 3.1 整体架构

```
┌─────────────────────────────────────┐
│         编排层（可多选）              │
│   ┌─────────┐  ┌─────────┐         │
│   │Leader A │  │Leader B │  ...    │
│   └────┬────┘  └────┬────┘         │
│        │            │               │
│   任务分配策略 ←→ 选举/切换           │
└────────┼────────────┼───────────────┘
         │
    ┌────┴────┐
    │ 专家池  │
    │ A B C  │ ← 每个有专长、评分、健康状态
    └─────────┘
```

### 3.2 核心组件

| 组件 | 职责 |
|------|------|
| **编排者（Orchestrator）** | 懂业务、会拆任务、能选协作模式 |
| **任务分配器** | 线性/树形/网状/流水线 可配置 |
| **专家 Agent 池** | 各有所长、只做擅长的 |
| **信息凝练层** | 压缩上下文，传递摘要而非原始数据 |

## 4. 协作模式

### 4.1 协作拓扑可配置

| 任务类型 | 适合的协作模式 |
|---------|---------------|
| 简单 bug 修复 | 线性（编排者 → 1 个 Agent） |
| 新功能开发 | 树形（拆成前端/后端/测试 并行） |
| 架构设计讨论 | 网状（多个 Agent 碰撞） |
| 多服务集成 | 流水线（A → B → C） |

### 4.2 设计要点

- 编排者是"元认知"的：不仅拆任务，还懂得"这个任务用什么姿势"
- 协作拓扑像 DAG 一样可声明配置
- 模式即代码：用配置或 DSL 声明，不是写死

## 5. 资源管理

### 5.1 现实问题映射

| 现实问题 | Agent 系统对应 |
|---------|---------------|
| 人员流动 | Agent 上下线（注册/发现/健康检查） |

### 5.2 设计要点

- Agent 注册与发现机制
- 能力边界声明与动态评估
- 负载均衡与任务分发

## 6. 考核与去中心化

### 6.1 中心化问题

> 这是一个纯中心化系统，产出好坏极大依赖编排者的能力和瓶颈

### 6.2 参考分布式算法的解决思路

| 问题 | 参考思路 |
|------|---------|
| 编排者挂了 | Raft Leader 选举 |
| 多个编排者谁更强 | 性能评估 + 动态选择 |
| 怎么考核 Agent 产出 | 交叉验证 + 历史成功率 + 人工反馈 |
| 怎么防止编排者摆烂 | OKR 拆解 + 结果审计 |

### 6.3 Raft 类算法的应用

- **Leader 选举**：多个编排者候选，谁更适合当前任务
- **Term 机制**：任务周期，完成后重新选举
- **日志复制**：任务状态持久化，不依赖单一节点
