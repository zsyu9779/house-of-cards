## 0. 背景与问题

### 0.1 HOC 当前的记忆盲区

v0.3 的 HOC 是一个**无状态调度器**——Speaker 规划 Bill、Whip 分发执行、Minister 处理任务，但每次 Session 都是"白纸一张"：

- Speaker 拆解 Bill 时不参考历史经验——同样的复杂任务每次都要重新评估
- Minister 匹配 Bill 只看 portfolio skill——不参考该 Minister 历史上在相似任务上的表现
- 错误会重复——Minister 在不同 Session 中踩相同的坑，系统无记忆
- Gazette 和 Hansard 记录了所有信息，但缺乏"提取有用信息"的机制

VatBrain 的设计哲学——"遗忘是默认，记住是例外"——恰恰是解决这个问题的钥匙。

### 0.2 集成定位

**不让 HOC 依赖 Docker**，不让两个项目的代码互相耦合。

方案：VatBrain 先完成自身轻量化改造（SQLite 后端），然后 HOC 通过 `go get` 引入作为普通 Go 库，零新进程依赖。

## 1. 集成架构

### 1.1 整体关系

```
┌────────────────────────────────────────────────────────────┐
│                      House of Cards                         │
│                                                             │
│  Speaker ──search──┐                                        │
│  (拆解Bill前查经验) │                                       │
│                    │     ┌──────────────────────────────┐  │
│  Minister ─write──┼─────┤  VatBrain (Go library)        │  │
│  (执行中记情境)    │     │                               │  │
│                    │     │  WeightDecay / SigGate        │  │
│  Whip ──touch─────┤     │  PatternSep / Retrieval       │  │
│  (完成后标记反馈)  │     │  Consolidation                │  │
│                    │     │                               │  │
│  CLI ──search─────┘     │  MemoryStore (interface)       │  │
│  (用户查询历史)          │    └── SQLite backend         │  │
│                          └──────────────────────────────┘  │
│                                                             │
│  存储：HOC 的 SQLite DB + VatBrain 的 SQLite (同文件/分表) │
└────────────────────────────────────────────────────────────┘
```

### 1.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| VatBrain 集成方式 | Go library import | 零网络调用、零外部进程、编译时检查 |
| VatBrain 存储后端 | SQLite | 与 HOC 现有存储一致，复用同一 DB 文件 |
| DB 文件 | 默认同一 `~/.hoc/hoc.db` | v0.4 不加新文件，VatBrain 表前缀独立 |
| 向量/Embedding | v0.4 不引入 | 用关键词匹配 + 结构化过滤替代，够用 |
| 睡眠整合触发 | Whip tick 末尾触发 | 复用现有调度循环，不加新定时器 |

## 2. HOC 侧的改动范围

### 2.1 新增 `internal/memory/` 包

包装 VatBrain 的初始化和调用，HOC 其他模块不直接 import VatBrain：

```
internal/memory/
├── memory.go          // Init(cfg) → *Client，包装 VatBrain
├── episodic.go        // WriteEpisodic — 情境记忆写入
├── search.go          // Search — 两阶段检索
├── feedback.go        // Feedback — 行为反馈（touch/write/update）
├── consolidation.go   // TriggerConsolidation — 睡眠整合触发
└── memory_test.go
```

### 2.2 调用点分布

| 调用点 | 操作 | 时机 | 优先级 |
|--------|------|------|--------|
| `internal/speaker/speaker.go` | `Search` | 拆解 Bill 前，查询历史上相似 Bill 的拆解经验 | P0 |
| `internal/speaker/speaker.go` | `WriteEpisodic` | Bill 拆解完成后，记录本次拆解+后续结果 | P0 |
| `internal/whip/scheduler.go` | `Search` | `matchBillToMinister` 时，查询 Minister 在同类 Bill 上的历史 | P1 |
| `internal/whip/liveness.go` | `WriteEpisodic` | `byElection` 触发时，记录 Minister 失败的情境 | P0 |
| `internal/whip/dispatch.go` | `WriteEpisodic` | Gazette 投递成功/失败时，记录投递结果 | P1 |
| `internal/whip/tick.go` | `TriggerConsolidation` | Session 结束后触发整合（提炼规则） | P2 |
| `cmd/bill.go` | `Search` | `hoc bill list --filter` 支持记忆搜索 | P2 |
| `internal/privy/` | `Search` | 合并仲裁时查询历史冲突处理经验 | P2 |

### 2.3 CLId 增强（D-3 + v0.4 新能力）

```bash
# 查询历史经验
hoc memory search "Redis connection pool exhausted"
hoc memory search --minister minister-backend --task-type debug
hoc memory search --session sprint-q1

# 手动触发整合
hoc memory consolidate --session sprint-q1

# 查看记忆统计
hoc memory stats
```

### 2.4 数据库变更

VatBrain 库在 SQLite 中创建独立表（前缀 `vb_`），与 HOC 表共存：

```sql
-- VatBrain 管理的表（由 VatBrain 库自动创建）
-- vb_episodic_memories  : 情境记忆
-- vb_semantic_memories  : 语义记忆（整合产物）
-- vb_memory_edges       : 记忆间关系边
-- vb_consolidation_runs : 整合运行记录

-- HOC 现有表不变
-- bills, ministers, sessions, gazettes, hansards, events, ...
```

### 2.5 配置文件变更

```toml
# configs/hoc.toml 新增
[memory]
enabled = true                  # 是否启用记忆系统
db_path = ""                    # 默认共用 hoc.db，可指定独立文件
consolidation_auto = false      # v0.4 默认手动触发整合

[memory.decay]
lambda = 0.1
alpha = 0.005
beta = 0.05
cooling_threshold = 0.01

[memory.gate]
min_cross_cycle = 2
min_subsequent_refs = 2
```

## 3. 核心场景设计

### 3.1 Speaker：规划时查经验

**场景**：Speaker 收到一个 Bill "实现 JWT 认证"，在拆解成子 Bill 之前，先搜索历史记忆。

```
Speaker.draftBill("实现 JWT 认证")
  │
  ├─ memory.Search(ctx, SearchRequest{
  │     Query:   "实现 JWT 认证",
  │     Context: {ProjectID: session.ProjectID, Language: "go", TaskType: "feature"},
  │     TopK:    5,
  │   })
  │
  ├─ 返回：
  │   [1] "JWT 认证需处理 token 刷新逻辑，上次忘了导致返工" (weight: 3.2)
  │   [2] "go-jwt 库 v5 API 有 breaking change" (weight: 2.1)
  │   [3] "认证中间件必须放在 CORS 中间件之后" (weight: 1.8)
  │
  └─ 拆解时注入上下文：
       Bill brief 中包含 "历史经验：注意 token 刷新、go-jwt v5 API"
```

**在 CLAUDE.md 中的体现**：Speaker 的 system prompt 中注入检索到的记忆摘要。

### 3.2 Minister：执行中记情境

**场景**：Minister 完成 Bill 后，将执行过程的关键节点写入记忆。

```
Minister.doWork(bill)
  │
  ├─ 遇到错误 "Redis MaxOpenConns=50 导致连接池耗尽"
  │   memory.WriteEpisodic(ctx, WriteRequest{
  │     Summary:   "Redis MaxOpenConns 默认 50 在生产环境不足",
  │     TaskType:  "debug",
  │     EntityID:  "func:NewRedisPool",
  │     IsCorrection: true,  // 这是纠正性经验
  │   })
  │
  ├─ 问题解决后
  │   memory.Feedback(ctx, memID, FeedbackAction{Action: "confirmed"})
  │
  └─ Bill 完成，写完 done-file
      memory.Feedback(ctx, memID, FeedbackAction{Action: "used"})
```

**在 CLAUDE.md 中的体现**：Minister 的 system prompt 中增加记忆操作指示——"遇到意外错误时，调用 `write_memory`"

### 3.3 Whip：用记忆改进路由

**场景**：`matchBillToMinister` 不再只看 portfolio skill，还参考历史表现。

```
matchBillToMinister(bill)
  │
  ├─ 候选 Minister 列表（按 portfolio 匹配）
  │   [minister-A: skill=backend, skill=go]
  │   [minister-B: skill=backend, skill=go]
  │
  ├─ memory.Search(ctx, {
  │     Query: bill.Description,
  │     Context: {TaskType: "feature"},
  │   })
  │   返回：minister-A 在 3 个相似 Bill 上 2 次失败
  │         minister-B 在 2 个相似 Bill 上全部成功
  │
  └─ 排序调整：minister-B 优先于 minister-A
```

### 3.4 Whip：自动整合

**场景**：Session 完成后，Whip 触发一次整合，从本 Session 的情境记忆中提炼规则。

```
Session.complete()
  │
  └─ whip.tick()
       if session.AllBillsEnacted() && memory.NeedsConsolidation():
         result := memory.TriggerConsolidation(sessionID)
         // result.RulesPersisted: 3
         // result.AverageAccuracy: 0.85
         // → 3 条新规则进入 SemanticMemory
```

## 5. 实施路线

### 前置条件

- [ ] VatBrain 完成存储层抽象重构（`docs/v0.2/00-storage-refactor-draft.md`）
- [ ] VatBrain SQLite 后端可用、有测试覆盖
- [ ] VatBrain 作为 Go 库可 `go get` 引入

### Phase 1：VatBrain 最小接入（P0）

```

  → memory.go：Init(VatBrain SQLite backend)
  → episodic.go：WriteEpisodic 封装
  → search.go：Search 封装
  → Speaker + Whip liveness 调用点集成
```

**验收**：
- [ ] `hoc doctor` 新增 memory 检查项
- [ ] Speaker 规划时可检索历史记忆
- [ ] by-election 触发时写入情境记忆

### Phase 2：记忆闭环（P1）

```
PR-2: 反馈 + 路由增强
  → Whip dispatch 写入投递结果
  → matchBillToMinister 集成记忆排序

```

**验收**：
- [ ] Minister 执行完成后，记忆有反馈更新

### Phase 3：用户可见 + 整合（P2）

```
PR-3: CLI + 自动整合
  → hoc memory search / stats / consolidate

```

**验收**：
- [ ] `hoc memory search` 可用
- [ ] 整合运行成功，提炼出语义规则
- [ ] `hoc bill list --filter` 支持记忆增强排序

## 6. 风险与待定项

| 风险 | 缓解 |
|------|------|
| VatBrain v0.2 延迟 | HOC 先 merge Phase 1，用 stub memory client（返回空结果），等 VatBrain 就绪后切 |
| 记忆噪声（无关记忆干扰决策） | 显著性门控 + 冷却阈值自然过滤低价值记忆 |
| LLM 提炼规则质量 | v0.4 先用简单模式（字符串拼接），v0.5 接 LLM |
| 单 SQLite 文件锁竞争 | VatBrain 写操作为异步批量写入，和 HOC 的 Whip tick 错开 |

### 待定项

- 记忆是否需要跨 HOC 项目共享（全局 ~/.hoc/memory.db vs 项目级 hoc.db）
- Minister 是否有权限限制（不能搜索其他 Minister 的记忆）
- 整合触发时机（Session 结束 vs 定时 vs 闲置检测）

## 7. 与 VatBrain 项目的分工

| 关注点 | HOC v0.4 负责 | VatBrain v0.2 负责 |
|--------|-------------|-------------------|
| 存储层抽象 | 不关心（用 MemoryStore 接口） | 定义接口 + SQLite 实现 |
| 核心算法 | 不修改 | 所有权重/门控/整合算法 |
| 调用封装 | `internal/memory/` 包装 | 不感知 HOC |
| CLAUDE.md 模板 | Speaker/Minister memory 指令 | 不感知 HOC |
| 测试 | 集成测试（HOC 场景） | 单元测试（算法 + 存储） |
| MCP 集成 | 不引入（HOC 不对外暴露 MCP） | VatBrain 自己的 MCP server |
